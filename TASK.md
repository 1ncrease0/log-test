# Тестовое задание «Парсер логов»
## 1 Общая цель
Разработать микросервис на Go, который принимает файлы-логи, парсит их и агрегирует

информацию, сохраняет полученные данные в PostgreSQL и предоставляет REST-API для

получения информации о сущностях.

Технологический минимум:

• язык — Go 1.22+ (можно использовать только стандартный пакет net/http);

• база — PostgreSQL (запуск через Docker-compose);

## 2 Функциональные требования
№	Функция	Что должно делаться
- F-1	Работа с файлом	Доступ к файлу осуществляется через монтированнуюпапку в самом проекте, название папки должно быть data/
- F-2	Парсинг	•  разбить файлы на секции и собрать информацию по уникальным идентификаторам в объекты•  не все объекты могут быть полнымипри любой ошибке отклонить файл и выкинуть ошибку.
- F-3	Агрегация топологии	На основе nodes и ports построить граф:•  узлы (host / switch);•  порты;будет плюсом если подумать какие связи могли бы быть, записать в Readme;
- F-4	Сохранение в БД	Таблицы (минимум): logs, nodes, ports, nodes_info. Все связи реализованы через FK.Также необходим скрипт создания баз данных либо же наличие миграций. Они должны применятьсяавтоматический при сборке.
- F-5	API-доступ к топологии	1.  POST /api/v1/parse/ – возвращает log_id. В тело передать путь до архива с логом2.  GET /api/v1/topology/{log_id} – возвращает список узлов и групп топологии.3.  GET /api/v1/node/{node_id} – детали узла системы из лога.GET/api/v1/port/{node_id} –получение портов узла системы из лога. GET /api/v1/log/{log_id} – мета-информация о логе (status,количествоузлов/портов,датазагрузки). 
- F-6	Контейнеризация	Docker-compose с сервисами app и db. Порт приложения –8080(можнопереопределитьпеременнойPORT).

## 3 Нефункциональные требования

Требование	Описание
Код-стайл	Go-fmt, lint-проверка.
Логирование	Структурные логи в stdout (запрос, время парсинга, ошибки).
Настройки	Через  переменные  окружения (DATABASE_URL, PORT, LOG_LEVEL).
Документация	README.md – как собрать, как запустить, примеры curl.
Отказоустойчивость	В случае битых логов нужна обработка ошибок.
## 4 Как сдавать

1\. Создайте репозиторий, сделайте его публичным.

2\. В корне разместите README.md и docker-compose.yml.

3\. Убедитесь, что docker compose up -d поднимает сервис без ошибок.

4\. Будет плюсом создание postman\_collection с тестами API

5\. Пришлите ссылку на репозиторий.
